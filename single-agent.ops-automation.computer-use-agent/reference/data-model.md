# Data model — computer-use-agent

Authoritative; `/akka:implement` produces these records and events verbatim.

| | `executedAt` | `Instant` | no | When the action was recorded on the entity. |

| `Task` (entity state) | `taskId` | `String` | no | — |
| | `request` | `Optional<TaskRequest>` | yes | Populated after `TaskSubmitted`. |
| | `actionHistory` | `List<ActionRecord>` | no | Append-only; empty list in initial state. |
| | `pendingConfirmation` | `Optional<ConfirmationRequest>` | yes | Non-empty while `AWAITING_CONFIRMATION`; cleared by `ConfirmationReceived`. |
| | `outcome` | `Optional<TaskOutcome>` | yes | Populated after `TaskCompleted`, `TaskHalted`, or `TaskFailed`. |
| | `status` | `TaskStatus` | no | See enum. |
| | `haltRequested` | `boolean` | no | Set to `true` by `TaskHalted`; never reset. |
| | `createdAt` | `Instant` | no | When `TaskSubmitted` emitted. |
| | `finishedAt` | `Optional<Instant>` | yes | Terminal-state timestamp. |

Every nullable field on `Task` is `Optional<T>`. The view's table updater wraps values with `Optional.of(...)`; callers use `.orElse(...)` or `.isPresent()` (Lesson 6).

## Enums

`ActionType`: `SCREENSHOT`, `CLICK`, `TYPE`, `SCROLL`, `KEY_PRESS`, `NAVIGATE`, `FILE_READ`, `FILE_WRITE`, `FILE_DELETE`, `FORM_SUBMIT`, `ACCOUNT_MODIFY`, `API_CALL`.

`ActionStatus`: `ALLOWED`, `BLOCKED`, `AWAITING_CONFIRMATION`, `CONFIRMED`, `REJECTED`.

`OutcomeStatus`: `SUCCESS`, `PARTIAL`, `FAILED`, `HALTED`.

`TaskStatus`: `SUBMITTED`, `RUNNING`, `AWAITING_CONFIRMATION`, `COMPLETED`, `HALTED`, `FAILED`.

## Events (`TaskEntity`)

| Event | Payload | Transition |
|---|---|---|
| `TaskSubmitted` | `request` | → SUBMITTED |
| `TaskStarted` | — | → RUNNING |
| `ActionProposed` | `toolCall` | (no status change; records the raw proposal before guardrail evaluation) |
| `ActionExecuted` | `actionRecord` | (no status change; appends to `actionHistory`) |
| `ConfirmationPending` | `confirmationRequest` | → AWAITING_CONFIRMATION |
| `ConfirmationReceived` | `confirmationResponse` | → RUNNING (clears `pendingConfirmation`) |
| `TaskCompleted` | `outcome` | → COMPLETED (terminal happy) |
| `TaskHalted` | `reason: String` | → HALTED (terminal) |
| `TaskFailed` | `reason: String` | → FAILED (terminal) |

`emptyState()` returns `Task.initial("")` with `request = Optional.empty()`, `actionHistory = Collections.emptyList()`, `pendingConfirmation = Optional.empty()`, `outcome = Optional.empty()`, `status = SUBMITTED`, `haltRequested = false`. `emptyState()` never references `commandContext()` (Lesson 3).

## View row

`TaskRow` mirrors `Task` minus `ScreenshotAttachment.imageBytes` — the screenshot bytes are served from `EnvironmentSimulator`'s in-memory cache, not from the view. The UI fetches the current screenshot via a dedicated endpoint if needed, but in the embedded sample it is passed inline as a base64 data URL on the `action-executed` SSE sub-event.

The view declares ONE query: `getAllTasks: SELECT * AS tasks FROM task_view`. No `WHERE status = :status` filter — Akka cannot auto-index enum columns (Lesson 2); the endpoint filters client-side.

## Task definition (`TasksTasks.java`)

```java
public final class TasksTasks {
  public static final Task<ToolCall> EXECUTE_TASK = Task
      .name("Execute task")
      .description("Observe the current screen and return the next ToolCall to execute, or return a TaskOutcome if the task is complete or unrecoverable")
      .resultConformsTo(ToolCall.class);

  private TasksTasks() {}
}
```

The companion `Tasks` class is mandatory for any `AutonomousAgent` (Lesson 7).
