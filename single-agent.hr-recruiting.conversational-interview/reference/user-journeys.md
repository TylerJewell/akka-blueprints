# User journeys — conversational-interview

## J1 — Complete a Software Engineer session end-to-end

**Preconditions:** Service running on declared port (`http://localhost:9427/`); a valid model-provider API key set, or the mock LLM selected at scaffold time.

**Steps:**
1. Open `http://localhost:9427/` → App UI tab.
2. From the **Role** dropdown, pick `Software Engineer (Backend)`.
3. Enter `candidate-01` in the Candidate handle field, enter your identifier in the Started by field.
4. Click **Start session**.
5. The session card appears in the live list in `OPEN` state. Within ~10–30 s, it transitions to `CONDUCTING` and the first question appears in the right pane (targeting `swe-problem-solving`).
6. Enter a candidate answer in the Answer textarea and click **Submit answer**.
7. The card transitions through `ANSWER_RECEIVED` → `ANSWER_SCREENED` → `CONDUCTING`. The next question appears.
8. Repeat until the session reaches `COMPLETED`.

**Expected:**
- The session reaches `COMPLETED` with one `TurnRecord` per conducted turn. The turn count equals `targetTurnCount` or fewer if the agent signals early completion.
- Every `TurnRecord` has a non-blank question, a `competencyId` that maps to a role competency, and a submitted and screened answer for every answered turn.
- The completion summary is visible in the right pane.

## J2 — Protected-category attribute in a candidate answer is screened

**Preconditions:** Service running. Any model provider (real or mock — the sanitizer runs either way).

**Steps:**
1. Start a new session using the `Product Manager` seeded role.
2. In the first answer, include the sentence: "I have a chronic condition that sometimes affects my energy levels, but I manage it well."
3. Submit the answer.

**Expected:**
- The card transitions to `ANSWER_SCREENED`. The right pane shows the screened answer with `[REDACTED-HEALTH]` in place of the disclosure sentence, and a health-category chip in the screening result.
- The subsequent question generated by the agent does not reference health conditions, chronic illness, or energy levels.
- The session's `turns[n].submittedAnswer.rawAnswer` in `GET /api/sessions/{id}` retains the original text; `turns[n].screenedAnswer.redactedAnswer` shows the redacted form.

## J3 — Guardrail blocks a prohibited question draft

**Preconditions:** Service running with the mock LLM selected (`model-provider = mock`). The mock's `conduct-turn.json` includes a deliberately malformed entry with a question that asks about family planning.

**Steps:**
1. Start sessions until the 4th session is created (the mock is seeded to produce a malformed entry on the first iteration of the 4th session).
2. Watch the first `conductTurnStep` lifecycle in the network panel (`/api/sessions/sse`).

**Expected:**
- The first agent iteration produces a question containing prohibited language.
- The `before-agent-response` guardrail rejects it. The prohibited question NEVER appears in the UI or in `InterviewSessionEntity` — there is no `TurnConducted` event with the prohibited payload.
- The agent loop retries on iteration 2 and produces a compliant question. The card transitions to `CONDUCTING` with a question that passes all four guardrail checks.
- The service log shows one `guardrail.reject` line per rejected iteration with the structured-error code naming which check failed.

## J4 — Audit log retains raw answer; view shows screened form

**Preconditions:** Service running with `LOG_LEVEL=DEBUG`. Any model provider.

**Steps:**
1. Start a session. In the first answer, include: "I follow Islam and observe Ramadan, so I'd need flexibility during that period."
2. Wait for `ANSWER_SCREENED`.
3. Fetch `GET /api/sessions/{id}`.

**Expected:**
- `turns[0].submittedAnswer.rawAnswer` in the JSON retains the original sentence with the religious reference intact.
- `turns[0].screenedAnswer.redactedAnswer` replaces the religious content with `[REDACTED-RELIGION]`.
- `turns[0].screenedAnswer.protectedCategoriesFound` lists `religion`.
- The SSE stream and the UI transcript show only the screened form.

## J5 — Off-competency question is rejected by the guardrail

**Preconditions:** Service running with mock LLM. The mock includes an entry whose `competencyId` is `"nonexistent-competency"` for a role that does not declare it.

**Steps:**
1. Start a session using the `Customer Success` seeded role.
2. Trigger the mock session index that returns the off-competency entry (see `src/main/resources/mock-responses/conduct-turn.json`).

**Expected:**
- The guardrail rejects the turn because `competencyId` is not in the role definition.
- The agent loop retries and returns a compliant turn with a valid `competencyId`.
- No `TurnConducted` event with the invalid `competencyId` appears in the entity log.

## J6 — Full transcript integrity after session complete

**Preconditions:** Service running. Any model provider.

**Steps:**
1. Complete a full session using the `Customer Success` role (4 competencies, `targetTurnCount = 4`).
2. Fetch `GET /api/sessions/{id}` after the session reaches `COMPLETED`.

**Expected:**
- `turns` has exactly 4 (or more, if follow-up turns were generated) `TurnRecord` entries.
- Every turn has a non-null `turn` (question), a non-null `submittedAnswer`, and a non-null `screenedAnswer`.
- Every `turn.competencyId` maps to one of the 4 declared competencies.
- `completedAt` is non-null and `status` is `COMPLETED`.
